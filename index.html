★ 3行まとめ

  • スキルとは＝「いつ・何をするか」を書いた SKILL.md と、必要なスクリプト／参照資料をまとめたフォルダ単位の能力パッケージ。
  • キモは progressive disclosure（段階的開示）＝起動時は名前と説明（約100トークン）だけ読み込み、タスクに合致した時だけ本文・付属ファイルを展開するので、コンテキストを節約しながら専門能力を持たせられる。
  • 使い分け＝常時の前提は CLAUDE.md、外部システムへのリアルタイム接続は MCP、「再利用したい手順・専門知識」はスキル。3つは競合せず補完関係。

01スキル（Agent Skills）とは何か

Agent Skills（エージェントスキル、略して「スキル」）とは、特定の作業手順・専門知識・関連リソース（スクリプトやテンプレート、参照ドキュメント）を ファイルシステム上のフォルダにパッケージ化して、Claude に「専門能力」を後付けする仕組みです。汎用アシスタントである Claude を、PDF処理の専門家・自社業務の担当者・特定コードベースの熟練者へと変身させます。

スキルは Agent Skills のオープン仕様に基づいており、Claude Code・API・Claude.ai など複数の環境で共通の形式が使えます。

核心となる考え方：Progressive Disclosure（段階的開示）

スキル最大の特徴は、情報を一度に全部読み込まず、必要になった段階で順に開示する3層構造です。これによって「分厚いマニュアルを丸ごと持たせると重い」という問題を回避します。

Lv.1
メタデータ（名前＋説明）セッション起動時に全スキル分が system prompt に常駐。Claude は「このタスクにこのスキルが要るか」を説明文だけで判断する。
≈ 100 tok / スキル
Lv.2
本文（SKILL.md の手順）スキルが「発火」した時だけ読み込まれる、具体的な手順・ガイダンス。
通常 < 5k tok
Lv.3+
付属リソース（参照ファイル・スクリプト・データ）SKILL.md から参照された時だけ読む／実行する。スクリプトは「コード本体」ではなく「出力」だけがトークンを消費する。
実質ほぼ無制限

結果として、スキルには大量のAPIリファレンスやデータセットを同梱しても、実際にアクセスするまでトークンを消費しません。これがスキルが「軽いのに物知り」でいられる理由です。

SKILL.md の構造

すべてのスキルは SKILL.md を必須とし、先頭の YAML フロントマター（メタデータ）＋ Markdown 本文（手順）で構成されます。

---
name: processing-pdfs
description: Extract text and tables from PDF files, fill forms, merge
  documents. Use when working with PDF files or when the user mentions
  PDFs, forms, or document extraction.
---

# PDF Processing

具体的な手順・例・ワークフローを Markdown で記述...

必須フィールド

フィールド制約役割・ポイント
name最大64文字／小文字・数字・ハイフンのみ／anthropic claude は予約語で不可スキルの識別子。動名詞（〜ing）形が推奨：processing-pdfs, analyzing-spreadsheets
description最大1024文字／空不可／三人称で記述最重要。Claude が本文を読む前に見る唯一の情報。「何をするか」＋「いつ使うか」を具体的に書く（後述）

このほか Claude Code の拡張として allowed-tools（使用可能ツールの制限）や disable-model-invocation: true（自動発火を無効化し明示呼び出し専用にする）などのオプションがあります。

フォルダ構成

スキルは「フォルダ」です。SKILL.md だけが必須で、それ以外は任意。慣習的なサブフォルダは次の通りです。

processing-pdfs/ ├── SKILL.md # 必須：手順 + YAMLフロントマター ├── scripts/ # 任意：実行コード(Python/Bash)。出力だけが文脈に載る │ ├── analyze_form.py │ └── fill_form.py ├── references/ # 任意：必要時だけ読む詳細ドキュメント │ ├── FORMS.md │ └── REFERENCE.md └── assets/ # 任意：出力に使うテンプレート・アイコン等 └── template.docx

02どう動くのか（技術的な仕組み）

発見とロードの流れ

  1. 起動時：Claude は利用可能な全スキルの YAML から「名前＋説明」だけを読み、system prompt に載せる。
  2. 会話中：ユーザーの要求とスキルの description を照合し、「このスキルが該当する」と判断する。
  3. 発火時：Bash／Read 系ツールでファイルシステムから SKILL.md 本文を読み込む。
  4. 必要に応じて：本文が参照する references/REFERENCE.md を読み、scripts/fill_form.py を実行する——タスクが要求した分だけ。

ファイルシステム＋コード実行が土台

スキルはコード実行環境（ファイルシステム＋Bash）の上で動きます。重要なのはスクリプトは「読む」のではなく「実行する」点です。SKILL.md に「analyze_form.py を実行せよ」と書けば、Claude は python scripts/analyze_form.py input.pdf を実行し、スクリプトのコード本体ではなく、その出力だけがコンテキストを消費します。決定的（deterministic）な処理をコードに任せることで、確実性とトークン効率を両立します。

コンテキスト効率の具体例： 参照ドキュメント10本を同梱したPDFスキルに対し、ユーザーが「テキストを抽出して」と頼むと、Claude は SKILL.md は読みますが、フォーム記入ガイドやリファレンスは読みません。消費トークン＝SKILL.md＋会話分のみ。スキル全体ではありません。

サーフェス間の同期はされません。 Claude Code に置いたスキルは API や Claude.ai に自動では現れません。後述のように Git をソース・オブ・トゥルースにして各環境へ配る運用が現実的です。

API で利用する場合は、ベータヘッダ（例：code-execution-2025-08-25, skills-2025-10-02, files-api-2025-04-14）が必要で、1リクエストあたり最大8スキルまでという制限があります。

04スキルの作り方

置き場所

# 個人用（全プロジェクトで有効）
~/.claude/skills/my-skill/SKILL.md

# プロジェクト用（そのリポジトリ内のみ。チームでGit共有しやすい）
./.claude/skills/my-skill/SKILL.md

作成の流れ

  1. 反復パターンを見つける：毎回同じ指示を貼り付けている、と気づいたらそれがスキル候補。
  2. mkdir -p ~/.claude/skills/skill-name でフォルダ作成。
  3. SKILL.md にフロントマター＋手順を書く。
  4. 必要なら references/ scripts/ assets/ を追加。
  5. /skill-name で手動起動、または自動発火を確認してテスト。
  6. 実際の使われ方を見て改善（イテレーション）。

ゼロから書く必要はありません。 Claude Code に「このプロジェクト用のデプロイ手順をスキルにして」と頼めば、適切なフロントマターと構成で SKILL.md を生成してくれます。Anthropic が配布する skill-creator（スキルを作るスキル）も利用できます（→ 09章）。

説明文（description）の書き方——ここが成否を分ける

description はスキルが発見される唯一のチャンスです。「何をするか（動作）」＋「いつ使うか（トリガー条件）」を、三人称・具体的に書きます。

✅ 良い例

"Extract text and tables from PDF files, fill forms, merge documents. Use when working with PDF files or when the user mentions PDFs, forms, or document extraction."

❌ 悪い例

"Helps with documents"（曖昧）／"Processes data"（いつ使うか不明）／"Does stuff with files"（不明確）

本文の構成（progressive disclosure を活かす）

  • SKILL.md 本文は500行以内を目安に。長い内容は別ファイルに分け、本文から参照する。
  • 参照は1階層まで：SKILL.md → reference.md はOK。SKILL.md → a.md → b.md → 本題 のような深いネストは、Claude が途中までしか読まないことがある。
  • パスは常にスラッシュ（scripts/helper.py）。Windows形式のバックスラッシュは不可。
  • ファイル名は説明的に（form_validation_rules.md。doc2.md はNG）。100行を超える参照ファイルには目次を付ける。
  • モデルごと（Haiku/Sonnet/Opus）に効きが違うので、複数モデルでテストする。Haiku ほど明示的な指示が必要。

API でカスタムスキルを作成

from anthropic.lib import files_from_dir
client = anthropic.Anthropic()

skill = client.beta.skills.create(
    display_title="Financial Analysis",
    files=files_from_dir("/path/to/financial_analysis_skill"),
)
print(skill.id, skill.latest_version)   # skill_01AbCd...

ルートに SKILL.md を含み、合計30MB未満であること。新バージョンは skills.versions.create() で作り、container 内で "latest" またはバージョン指定して利用します。

05呼び出し・利用方法

① 自動発火（デフォルト）

ユーザーの要求が description に合致すると、Claude が自動的にスキルを読み込んで実行します。コマンド入力は不要。

ユーザー: 「このPDFからテキストを抽出して要約して」
  ↓
Claude(内部): pdf スキルの説明に合致 → SKILL.md を読み込み → 抽出スクリプト実行

② 明示的に呼び出す

# Claude Code でスラッシュ呼び出し
/processing-pdfs

# プラグイン内スキルは名前空間付き
/my-plugin:hello

# 引数を渡す（SKILL.md内の $ARGUMENTS が受け取る）
/greeting Alex

③ API（container で指定）

container={
  "skills": [
    {"type":"anthropic", "skill_id":"pptx", "version":"latest"},
    {"type":"custom", "skill_id":"skill_01ABC...", "version":"latest"}
  ]
}

事前ビルド済みは短いID（pptx等）、カスタムは skill_01... 形式のIDで指定します。指定後は関連タスクで自動的に使われます。

06CLAUDE.md / MCP との違い

3つはよく混同されますが、役割が異なる補完的な機能です。「ロード方式」と「用途」で整理すると分かりやすいです。

機能置き場所ロード方式主な用途
CLAUDE.mdプロジェクトルート等セッション開始時に全文を読む（高い先行コスト）常に知っておくべき前提：コーディング規約・設計方針・チームの好み
Skills.claude/skills/ / API説明だけ常駐し、合致時に遅延ロード再利用したい手順・専門知識・複雑な多段ワークフロー
MCP外部サーバ / ローカルCLIツール名のみ常駐、スキーマは遅延（tool search）外部システムへのリアルタイム接続（GitHub・DB・Slack等）

CLAUDE.md を使う時

毎回必要な前提。「常にTypeScript」「このスタイルに従う」など、セッションを通じて一定のコンテキスト。

Skills を使う時

磨き上げた多段ワークフロー、説明の長い専門知識、プロジェクト横断で再利用したい手順。文脈を軽く保ちたい時。

MCP を使う時

ライブAPI接続が要る、頻繁に変わるデータ（issue・ログ・DB）、別アプリで開くようなサービス連携。

3つは一緒に使うのが理想： CLAUDE.md が「このプロジェクトはTS・RESTアーキ・Jest」と前提を示し、Skill が「BigQuery分析の手順＋テーブルスキーマ」を提供し、MCP が「GitHubのissueやライブDB」へ接続する。役割分担すれば文脈も軽く、能力も高くなります。スキルが MCP ツールを呼ぶ手順を書く、といった連携も可能です。

08利点と欠点・制限

利点

  • 専門能力化：汎用Claudeをドメイン専門家に変える。
  • 再利用性：一度作れば会話・プロジェクトをまたいで使える。
  • 文脈効率：段階的開示で文脈を軽く保つ。大量の参照資料を同梱しても使うまで無料。
  • 自動発見：タスクに応じてClaudeが適切なスキルを自動起動。
  • クロス環境：ファイルベースなのでCode/APIで共通の形式。

欠点・制限

項目内容
サーフェス間で非同期Code / API / Claude.ai 間で自動同期しない。Gitを原本に各環境へ配布する運用が必要。
共有範囲の差Claude.aiは個人のみ・組織管理なし。APIはワークスペース全体。Codeは個人/プロジェクト/プラグイン。
実行環境の制約API実行環境はネットワーク非接続・追加パッケージ導入不可（プリインストールのみ）。Claude.aiはネット接続が設定依存。
1リクエスト最大8スキル（API）超える場合は統合かルーティングが必要。
ZDR非対象Zero Data Retentionの対象外（標準保持ポリシー）。
ガバナンス／バージョン管理自動ロールバックや中央管理は組み込みでない。チェックサム・署名コミット等は自前で。

セキュリティ上の注意（重要）

スキルはコードを実行し、ツールにアクセスできます。信頼できないソースのスキルは「ソフトをインストールするのと同じ」リスクがあります。導入前に必ず監査してください。

  • 脅威の例：悪意あるスクリプト実行、安全規則を回避させる指示、認証情報のハードコード、パストラバーサル（../）、外部URLへのデータ送信。
  • 対策：信頼できる出所（自作 or Anthropic）のみ使う／SKILL.md・スクリプト・参照ファイルを全て事前レビュー／http curl fetch requests 等のネットワーク呼び出しを検索／秘密情報は環境変数や安全なストアで管理。

「スキルを作るスキル」＝ skill-creator

Claude 自身がスキル形式を理解しているため、新しいスキルを足場（scaffold）するメタスキルを使えます。Anthropic の skills リポジトリには template/ や skill 作成支援が含まれ、「ドメイン知識を再利用可能なスキルに整形する」用途で利用できます。これを使えば、適切なフロントマター・構成・ベストプラクティスに沿った SKILL.md をClaudeに生成させられます。

---
name: skill-creator
description: Generate a new Agent Skill by capturing domain knowledge.
  Use when creating a new skill or refactoring instructions into a reusable skill.
---
# Skill Creator — 適切なフロントマターと構成でSKILL.mdを足場生成する手順...

スキルを「探す」場所（マーケットプレイス）

  • anthropics/skills — 公式の参照実装・テンプレ・仕様。
  • Claude Code のプラグインマーケットプレイスから /plugin install で導入。
  • コミュニティへの提出は claude.ai/settings/plugins/submit / platform.claude.com/plugins/submit。安全審査＋コードレビューを通過したものが anthropics/claude-plugins-community カタログにコミットSHA固定で掲載される。
  • "awesome-claude-skills" 系の有志リポジトリも探索の入口になる（導入前のセキュリティ監査は必須）。

11このページを GitHub + Vercel で公開する

このガイドは単一の index.html（外部依存なし）として作られているので、そのまま静的サイトとしてホスティングできます。チームへ共有する手順は次の通りです。

A. GitHub にプッシュ

cd claude-skills-guide
git init && git add . && git commit -m "Add Claude Skills guide"
git branch -M main
git remote add origin git@github.com:<you>/claude-skills-guide.git
git push -u origin main

B. Vercel で公開（2通り）

  • ダッシュボード：Vercel で「Add New → Project」→ 当該リポジトリをImport → フレームワークは「Other」、Build/Outputは空のままDeploy。index.html がそのまま配信される。
  • CLI：npm i -g vercel の後、フォルダ内で vercel（プレビュー）／vercel --prod（本番）。

ルートに index.html があれば設定は不要ですが、リダイレクトやキャッシュを制御したい場合は vercel.json を追加できます。社内限定にしたい場合は Vercel の Password Protection や Vercel Authentication（チームプラン）を使うと、URLを知っていても許可者だけが閲覧できます。

公開後、Vercel が発行する https://<project>.vercel.app をチームメンバーに共有すれば閲覧できます。GitHub に push するたび自動で再デプロイされます。
