PART 1

1Skills とは

そもそもスキルって何？ どう動くの？ よく似た機能とどう違うの？——まずは「正体」を理解します。

1.1スキルってなに？

スキル（Agent Skills）とは、ひとことで言うと 「Claudeに特定の得意分野を覚えさせるための“手順書フォルダ”」です。「こういう作業のやり方」「この分野の知識」「使ってよい道具やテンプレート」を1つのフォルダにまとめて渡しておくと、必要なときにClaudeがそれを開いて、その分野の専門家のようにふるまってくれます。

たとえば、ふだんは何でも屋のClaudeを、PDFを扱うプロ・自社の書類づくり担当・あるソフトに詳しい技術者へと“変身”させられる、というイメージです。この仕組みは 公開された共通ルール（オープン仕様）に沿っていて、Claude Code・API・Claude.ai のどれでも同じ形式で使えます。Anthropic が2025年10月に正式発表しました（→ 発表記事）。

1.2仕組み：必要なときだけ開く

スキルの賢いところは、中身を最初から全部読むのではなく、必要になった分だけ順番に開く仕組みになっている点です（むずかしく言うと「progressive disclosure＝段階的開示」）。分厚いマニュアルを丸ごと暗記させると“重く”なってしまいますが、それを避けられます。

用語メモ：「トークン」はAIが一度に読める文章量の単位、「コンテキスト」はAIがいま見ている情報のまとまり（作業机の広さ）くらいに考えてください。スキルは、この“机”を散らかさずに専門知識を使えるのが利点です。

具体的には、次の3段階で「必要な分だけ」開いていきます。

1段目
名前と説明だけ覚える起動時、Claudeは各スキルの「名前」と「ひとこと説明」だけをサッと見ておく（中身はまだ読まない）。
≈ 100 tok / 個
2段目
合致したら中身を開くあなたの依頼が説明にぴったり合うと「出番だ」と判断し、手順書（SKILL.md）の本文を読み込む。
通常 < 5,000 tok
3段目
足りなければ追加で開く・道具を動かす参照ファイルを読んだり、付属プログラムを実行したり——その作業に必要な分だけ。
実質ほぼ無制限

この仕組みのおかげで、スキルに大量の資料やデータを同梱しても、実際に使うまでは“机”を圧迫しません。スキルが「軽いのに物知り」でいられる理由です。

付属プログラムは「読む」のではなく「動かす」

スキルには小さなプログラム（スクリプト）を同梱できます。ポイントは、Claudeがそれを文章として読むのではなく、実際に実行して“結果だけ”を受け取ること。たとえば「フォームを解析するプログラムを動かして」と書いておけば、Claudeはそれを実行し、出てきた答えだけを使います。長いプログラム本体は読まないので、机が散らからず、処理も正確になります。

1.3スキルの中身（SKILL.md とフォルダ）

スキルは「フォルダ」です。中で必ず必要なのは SKILL.md という1枚のファイルだけ。冒頭に「名前・説明」などの見出し情報、その下にやり方の手順を書きます。

---
name: processing-pdfs                # スキルの名前
description: PDFからテキストや表を取り出し、フォーム記入や結合を行う。
  PDFやフォーム、書類の抽出が話題に出たら使う。   # ← 「何を・いつ」を書く（最重要）
---

# PDF Processing

ここに具体的な手順・例・ワークフローを書く...

必ず書く2項目

項目役割（やさしく）ルール
nameスキルの名前（呼び出すときの目印）小文字・数字・ハイフンのみ／64文字以内。例：processing-pdfs
description最重要。Claudeが中身を読む前に見る唯一の手がかり。「何をする」＋「いつ使う」を書く1024文字以内（書き方は3.2で詳しく）

フォルダの中身（SKILL.md 以外は任意）

processing-pdfs/ ├── SKILL.md # 必須：名前・説明＋手順 ├── scripts/ # 任意：実行する小さなプログラム │ └── fill_form.py ├── references/ # 任意：必要なときだけ読む詳しい資料 │ └── FORMS.md └── assets/ # 任意：出力に使うテンプレートや画像 └── template.docx

この“フォルダごと”がスキルです。中身を見たい場合は、後半の公式スキル17種から実物を覗けます。

1.4似た機能との違い（CLAUDE.md / MCP）

名前が似ていて混乱しやすい3つですが、役割はハッキリ違います。まずはたとえ話でイメージをつかみましょう。

Claudeを“新しく来たスタッフ”にたとえると——

  • CLAUDE.md＝「社内のルールブック」。出社したら毎回まず目を通す、いつも守ってほしい決まりごと。
  • スキル＝「業務マニュアル集」。必要なときに棚から取り出して見る、特定作業のやり方。
  • MCP＝「社内システムへのアクセス権」。在庫DBやSlackなどにつないで、その場の最新データをやりとりするためのID。

もう少しきちんと整理すると、次のようになります。

機能いつ読み込まれる？何に使う？
CLAUDE.md会話のはじめに全文毎回守ってほしい前提：書き方ルール・方針・好み
Skills説明だけ常駐し、合致時に必要な分だけくり返し使う作業手順・専門知識
MCP道具名だけ常駐、使う時に詳細外部サービスとのリアルタイム接続（GitHub・DB・Slack等）

3つは一緒に使うのが理想： CLAUDE.md で前提を示し、スキル で作業手順を渡し、MCP で外部データにつなぐ——役割分担すれば、軽いまま賢く動けます。スキルから MCP の道具を呼ぶこともできます。

1.5メリットと注意点

うれしいところ

  • 専門家になれる：何でも屋のClaudeを、特定分野のプロに変身させられる。
  • 使い回せる：一度作れば、別の会話やプロジェクトでも使える。
  • 軽い：必要なときだけ開くので、たくさん資料を入れても重くならない。
  • 勝手に気づく：依頼の内容に合うスキルを、Claudeが自動で選んで使ってくれる。

気をつけたいところ

注意点内容
環境ごとに別管理Claude Code / API / Claude.ai の間で自動同期しない。場所ごとに入れ直しが必要。
共有範囲の違いClaude.aiは個人だけ。APIはチーム（ワークスペース）全体。Codeは個人/プロジェクト/プラグイン。
制限API利用では1回の依頼につき最大8スキルまで、など細かな制限あり。

セキュリティの注意：スキルはプログラムを実行できます。 知らない人が作ったスキルを入れるのは「出どころ不明のアプリをインストールする」のと同じリスクがあります。信頼できる出どころ（自分で作ったもの／Anthropic公式）だけを使い、入れる前に中身（SKILL.md・スクリプト）に目を通しましょう。心配なら「このスキル、安全か中身を確認して」とClaudeに頼むのも有効です。

2Skills を使う

既にあるスキルを実際に使ってみるパート。どこで使えて、どんな便利なスキルがあり、どう入れて、どう呼び出すか。

2.1どこで使える？

環境用意済みスキル自分のスキルの置き場共有範囲
Claude Code（CLI）プラグイン経由~/.claude/skills/（個人）／.claude/skills/（プロジェクト）個人・プロジェクト・チーム共有

2.3導入のしかた

いちばんかんたんなのは「Claudeに聞くこと」です。

下の手順がむずかしく感じても大丈夫。Claudeにふつうの言葉でこう頼んでみてください：

「ClaudeのスキルをこのPCに入れたいです。私の環境を確認しながら、手順を1つずつ教えてください」

するとあなたのパソコンの状態を調べて、必要な操作を1つずつ・コピペできる形で案内し、つまずいたら一緒に解決してくれます。以下は「中身を知りたい人」向けの参考です。読み飛ばしてOK。

A. Claude Code に入れる

方法① 公式リポジトリからコピー（確実）

# 公式スキル集を入手
git clone https://github.com/anthropics/skills.git

# 使いたいスキルを ~/.claude/skills/ にコピー（例：pdf と skill-creator）
mkdir -p ~/.claude/skills
cp -r skills/skills/pdf           ~/.claude/skills/
cp -r skills/skills/skill-creator ~/.claude/skills/

# Claude Code を再起動 → /pdf や /skill-creator が使える

方法② プラグインとして入れる

# スキル集（GitHub）をマーケットプレイスとして登録
/plugin marketplace add anthropics/skills

# プラグインを検索して導入
/plugin install <plugin-name>@anthropics-skills

B. Claude API で使う（開発者向け・Python）

import anthropic
client = anthropic.Anthropic()

resp = client.beta.messages.create(
    model="claude-opus-4-8",
    max_tokens=16000,
    betas=["code-execution-2025-08-25", "skills-2025-10-02"],
    tools=[{"type":"code_execution_20250825", "name":"code_execution"}],
    container={"skills":[
        {"type":"anthropic", "skill_id":"pdf", "version":"latest"},
    ]},
    messages=[{"role":"user", "content":"四半期レポートのPDFを作って"}],
)

用意済みスキルのIDは pdf docx pptx xlsx。自作スキルのアップロード方法は「チームに配る」で扱います。

C. Claude.ai（Web/アプリ）に入れる

  1. スキルのフォルダ（SKILL.md を含む）を zip圧縮。
  2. Claude.ai の 設定 → Capabilities / Features から Skills を開く（コード実行が使えるPro/Max/Team/Enterpriseプラン）。
  3. zipをアップロード。以降、関連する依頼で自動的に使われます。

2.4呼び出し方

① 自動で呼ばれる（基本）

あなたの依頼がスキルの説明に合えば、Claudeが自動でスキルを開いて使います。特別な操作は不要です。

あなた: 「このPDFからテキストを抜き出して要約して」
  ↓
Claude: pdf スキルが該当 → 自動で開いて抽出・要約

② 自分で指定して呼ぶ

# Claude Code でスラッシュ呼び出し
/processing-pdfs

# プラグイン内のスキルは「プラグイン名:スキル名」
/my-plugin:hello

2.5活用事例

📊 財務分析

社内データベースに問い合わせて売上や指標を集計。テーブルの構造や用語の決まりをスキルに同梱しておく。

📝 PDFフォーム記入

項目を読み取り→記入→確認まで自動化。記入ルールを別ファイルに分けて必要時だけ参照。

🔍 コードレビュー

構造・エラー処理・セキュリティを点検。GitHub(MCP)と組み合わせて実際のレビューにも。

📣 社内文書

状況報告や社内報を、自社のいつものフォーマットで作成（公式の internal-comms が好例）。

育て方のコツ： 最初は「売上レポート整形」のように小さく狭く作り、慣れてきたら関連スキルをまとめて「営業業務セット」のように育てるのがおすすめです。

PART 3

3Skills を作る・配る

自分だけのスキルを作るパート。難しく考えず、Claudeに手伝ってもらえばOK。作り方のコツとチーム共有を紹介します。

3.1作り方の流れ

作るのもClaudeにおまかせできます。 「こういう作業を覚えるスキルを作って」と頼めば、フォルダとファイルのひな形を作ってくれます。公式の skill-creator（スキルを作るスキル）を入れておくと、より本格的に作成・改善できます。

自分で作るときの基本の流れはこうです。

  1. くり返している作業を見つける：毎回同じ指示を出している＝スキル化のチャンス。
  2. フォルダを作る：~/.claude/skills/スキル名/
  3. SKILL.md を書く：名前・説明（次の3.2）＋手順。
  4. 必要なら資料やプログラムを足す：references/ scripts/ など。
  5. 試して直す：実際に使ってみて、説明文や手順を調整。

3.2説明文（description）の書き方

説明文は、スキルが使われるかどうかを左右する一番大事な部分です。Claudeはこの一文を見て「使うべきか」を判断します。コツは「何をするか」＋「いつ使うか」をセットで、具体的に書くこと。

✅ 良い例

「PDFからテキストや表を抽出し、フォーム記入や結合を行う。PDF・フォーム・書類の抽出が話題に出たら使う。」

❌ 惜しい例

「書類を手伝う」（ふんわりしすぎ）／「データを処理する」（いつ使うか不明）

そのほかのコツ：

  • 本文は長すぎない（目安500行以内）。詳しい内容は別ファイルに分けて、必要時だけ読ませる。
  • 別ファイルの参照は1段だけ（深い入れ子は読み落とされやすい）。
  • ファイル名は中身がわかる名前に（doc2.md ではなく form_rules.md）。

3.31分でひな形を作る

手を動かしてみたい方向けに、コピペで動く最小例です（Claude Code）。

mkdir -p ~/.claude/skills/release-notes
cat > ~/.claude/skills/release-notes/SKILL.md <<'EOF'
---
name: release-notes
description: gitのコミット履歴から、自社の体裁でリリースノートを作る。
  「リリースノート」「変更点まとめ」を頼まれたら使う。
---

# Release Notes Writer

1. `git log <前>..<今> --oneline` で対象コミットを集める
2. 新機能 / 修正 / 重要な変更 に分類する
3. Markdown で見やすくまとめて出力する
EOF
# Claude Code を再起動 → /release-notes で使える

API で自作スキルを登録したい場合は、フォルダごとアップロードします（開発者向け）。

from anthropic.lib import files_from_dir
skill = client.beta.skills.create(
    display_title="Release Notes",
    files=files_from_dir("./release-notes"),
)  # → skill.id をメッセージの container で指定して使う

3.4チームに配る・共有する

方法どんな仕組み？向いている相手
Gitで共有.claude/skills/ をリポジトリに入れて配る。変更履歴も残る同じリポジトリを使うチーム
プラグイン化マニフェストを付けて配布。/plugin install で導入組織全体・社外公開
社内マーケット社内GitHubにマーケット定義を置き /plugin marketplace add社内メンバー限定
APIで共有アップロードしてワークスペース全員で利用API利用のチーム

おすすめ運用： スキルの原本はGitで一元管理し、そこから各環境（Claude Code / API / Claude.ai）へ配る。こうすれば「どれが最新か分からない」を防げます。
